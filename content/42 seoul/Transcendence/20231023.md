# 학습의 갈피를 잡지 못하다
사실 이실직고하면 Javascript의 사용은 전에 경험 해본 적이 있었다. 이 블로그를 운영하게 된 시점인 2021년 말 ~ 2022년 초반에 기술 블로그를 운영하는 것이 중요하다는 것을 배우고, 짬짬히 글을 쓰기 위해 스터디를 하고, 배운 경험이 있었다. 

처음엔 그렇기에 다소 자만감?에 가까운 것이 있었다. 자바 스크립트에 대한 학습도 빠르게 진행되어 갔다. TS도 걱정은 되었으나, 생각보다 '약하지 않을까?' 내지는 '쉽지 않을까?' 라는 오만한 생각이 있었고, 그것을 깨닫는데는 그렇게 긴 시간은 걸리지 않았던 것 같다. 

## JS와 TS
객체 지향의 오리지날, C++를 배운 뒤, 그것도 아주 올드한 객체 지향을 42서울에서 배우면서 객체 지향 자체는 그렇게 '어렵다'고 생각하진 않았다. 개념적으로 다소 난해하거나, 추상적인게 있긴 했지만, 그게 무슨 역사나 사회 이론 외우는 것보단 훨씬 쉬운 이상 그렇게 어렵다고 생각은 하지 않았다. 

하지만 역시 세상은 그렇게 녹록하지 않은거다. 쉽다고 생각하면 어려우며, 어렵다고 생각하고 두려워하던게 막상 별거 아닌 경우도 너무나 많았고, 이런 너무나 뻔한 스테레오타입은 오늘도 1승을 거두었다. 

JavaScript의 구조는 우선 C++에서 생각하는 정도로 로우하지 않았다. 머릿속에서 딱딱 움직이는 비트 덩어리가 아니라는 지점은 계속해서 의문을 만들었고, 그걸 까보면 시간이 하루 그냥 날아가는 '기적'을 경험하게 될 때마다, '이게 맞나?' 라는 생각을 할 수 밖에 없었다. 

마지막 과제를 진행중이고, 이 과제를 언제까지 끝내야 한다는 은근한 압박. 거기서 오는 막연한 '딜레이 되는 것 같은 느낌'은 나를, 팀원들을 답답하게 만들기 시작했었다. 

뿐만 아니라 Javascript 를 제대로 배우려면 난해하니, 빠르게 배우자고 한 것에서부터 삐그덕 거렸는데, TypeScript까지 넘어가면서 더더욱 '왜?' 라는 것에 대해 이해할 수 있기 위해선 시간과 기꺼이 까서 집중이 필요했다. TypeScript에 가 자바 스크립트 기준, 더욱 다른 점은 다음과 같다고 볼 수 있다. 

1. **정적 타입 지정**: TypeScript는 변수, 함수의 매개변수, 객체 속성 등에 타입을 지정할 수 있게 해준다. 이는 코드의 안정성과 가독성을 높여준다. 

2. **강력한 객체 지향 프로그래밍 지원**: 클래스, 인터페이스, 상속 등 전통적인 객체 지향 개념을 지원한다.

3. **컴파일 시간 오류 검사**: TypeScript는 컴파일 시에 타입 오류를 포착하여, 런타임 오류를 줄일 수 있게 도와준다.

4. **네임스페이스와 모듈**: 코드를 더 체계적으로 조직화할 수 있는 네임스페이스와 모듈 시스템을 제공한다.

어디까지나 '좋게' 말하면 이렇게 차이점을 이야기 해볼 수 있는데, 막상 뒤집어 본다면 다음과 같은 특징을 가진다고 생각해볼 수 있다. 

1. 정적 타입을 지정해주는 만큼 변수 마다 타입을 고려해야하고, 객체 지향을 지향하는 만큼 타입을 제작하는 압박, 타입 구성에 있어서 문제가 없어야 한다는 문제를 가지게 된다. 
   
2. 객체 지향형을 중시하는 만큼, 수직적, 수평적 스케일의 확대, 더불어 C++ 대비 복잡하고 추상화된 기능들을 갖고 있는 만큼 독자적인 단어, 구문들은 러닝커브에 지대한 영향을 준다. 
   
3. 네임스페이스, 모듈 시스템이라고 이야기 하는 부분은 매우 유용하고, 이를 적극 활용하는 nestJS 의 경우 언어와 프레임워크 사이의 호흡이 아주 좋다. 그러나 반대로 말하면 언어를 배우고 바로 사용해야 하는 사람 입장에서는 이게 무슨 말인가? 라는 생각이 들게 된다.

## 앞으로 나가지 못하는 백-엔드 

![언발에 오줌 누기](https://media.giphy.com/media/nrXif9YExO9EI/giphy.gif)
> 으악 백엔드가 죽었어..!

내가 위의 문제들을 해결하고, 팀에 양해를 구하고 돌아와서 급하게 일을 진행할 생각이었다. 내 역할치가 얼마가 될진 모르겠지만, 서버 모듈 하나 구축하면 되지 않을까? 라는 생각을 했다. 그러나 막상 돌아온 상황은 뭔가 이상했다. 

이미 2개월 이상을 학습으로 사용한 상태였고, NestJS로 어떻게 설계하면 될지 다들 망설이고 있었기에, 가능한 클래스 구조를 최대한 활용하고, 특히 구현해야할 목표가 명확하기 때문에 이에 맞춰서 진행하면 되겠다고 생각하고 있었다. 

> NestJS의 특징 '모듈화'
> "NestJS는 애플리케이션을 여러 모듈로 분할합니다. 각 모듈은 관련된 기능들의 집합으로, 애플리케이션의 구성 요소를 체계적으로 관리할 수 있게 해줍니다." by ChatGPT 

그렇다. NestJS는 기본적으로 이제 와서 보면 Spring, Spring Boot를 통한 백엔드 서버 구축을 위한 구조와 매우 흡사하면서도, 좀더 불편하거나 딱딱하게 추상화된 내역을 풀어내놨고, 이런 점에서 상당히 '주무르기' 편리한 구조라고 볼 수 있겠다. 그렇기에 내가 생각한 내용은 다음과 같았다. 

1. User 모듈 : 인증과 User객체를 구축하고 있다. TypeORM으로 연결되어 User entity 를 만들고 UserRepository는 기본 기능들을 갖추고 있다. 
2. Chatting 모듈 : 내부에 Socket.IO 를 활용한 실시간 양방향성 메신저 룸이 구축되어야 하니, 소켓IO를 기반으로 room 당 message를 전달하기 유용한 기능별 클래스를 구축하자. 
3. Game  모듈 : Pong 게임을 구현해야 하니, 이를 위해선 공의 객체, 플레이어 객체들, 게임 자체의 기능이나 옵션들을 담은 객체 이렇게 3가지로 크게 분할하여 하위클래스를 구축하고, 이 역시 room 기반으로 진행하자- 

기능을 정리하고, 내용을 정리하고 보니 생각보다 해야할 일은 많지 않아 안심이었다. 구현할 목표에 맞춰 최소 단위를 room으로 설정했으며, room이라고 함은 Socket.IO에서 제공하는 양방향 통신이 담겨 있을 일정한 구역이었고, 이러한 내용을 배우고 난 뒤에서 생각했을 때 그렇게 어렵지 않을 것이라 생각하고 있었다.
